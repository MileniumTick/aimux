# aimux TUI — UI/UX analysis vs. crush (Charm)

**Scope:** concrete, ranked diff of UI/UX patterns `crush` (charmbracelet/crush) uses that `aimux` — a bubbletea v1 + lipgloss + huh **config manager**, NOT a chat app — could realistically adopt. Chat-specific machinery is explicitly skipped.

**Stack mismatch (matters for every recommendation below):**

- crush runs **bubbletea v2 + lipgloss v2 + ultraviolet** (Charm-internal, NOT public) + its own `dialog` system. It does **not** use `huh`.
- aimux runs **bubbletea v1 + lipgloss v1 + huh** + hand-rolled `View()` string composition.
- Therefore: nothing that depends on `ultraviolet` (ScreenBuffer / `uv.Screen` / `layout.Split`) is directly portable. What *is* portable is the **discipline** (precomputed layout rectangles, frame math, typed chrome), rebuilt with plain Go + lipgloss v1.

## ADOPT (ranked)

### A1 — Persistent footer = `help.Model` keybinding bar (highest ROI)

**Why now:** aimux already declares a full `keyMap` (`model.go:98-127`, `menuKeys`) with help strings, and the `?` overlay is wired in `Update` but **dead in render** — see Appendix. crush shows the correct pattern: a single always-visible help row driven by `bubbles/v2/help.Model`.

**crush source (evidence):**

- `model/status.go:11-33` — `Status` wraps `help.Model`; `Draw` (`status.go:89-115`) renders `s.help.View(s.helpKm)` into the fixed status rectangle.
- `model/ui.go:2420+` — `ShortHelp()` / `FullHelp()` return `key.Binding` slices that adapt to focus/state (e.g. `tab` help flips between "focus chat" / "focus editor"; `esc` shows "press again to cancel").
- `model/status.go:54-58` — `ToggleHelp()` flips short ↔ full; drawn into `layout.status` rectangle so it never scrolls with content.

**aimux target (evidence):**

- Help is duplicated as inline strings at the bottom of *each* view's content, not a fixed footer:
  - `table.go:91` → `helpStyle.Render("↑/↓ navigate · Enter = Switch · a add · d delete · e edit · r retry · t test · Esc back")`
  - `menu.go:60` → `helpStyle.Render("↑/↓ k/j · Enter · q quit")`
  - `model.go:1880` (`renderManageBindings`) → its own inline help string.
- These strings are part of `content`, so they scroll away and are inconsistent per view.
- **Dead code:** `renderHelpOverlay()` (`model.go:1990`) is defined but never called; `showHelp` is set at `model.go:412-419` and never read in `View()`. Pressing `?` does nothing visible.

**Effort:** **M.** Add a `help.Model` field to `model`, give it `menuKeys` (already implements nothing yet — make `keyMap` satisfy `help.KeyMap` via `ShortHelp`/`FullHelp`), render it into a fixed 1-line footer in `View()`, delete the per-view inline help strings. Resurrect or remove `renderHelpOverlay`/`showHelp`.

**Benefit:** one source of truth for keybindings; `?` toggles full help (and stops being a no-op); footer survives scrolling; consistent across every view including forms.

### A2 — Persistent header bar: app title + context breadcrumb

**Why now:** aimux has **no title on most views**. Only the dashboard shows an ASCII logo box (`model.go:1283-1300`), and the switch flow shows a dot stepper (`renderSwitchStepper`, `model.go:1962-2007`). `providerListView`, `manage bindings`, `restore`, every `huh` form — no header, no indication of where you are.

**crush source (evidence):**

- `model/header.go:95-150` — `drawHeader()` renders logo + compact metadata in one row.
- `model/header.go:155-205` — `renderHeaderDetails()` composes parts (`working dir • context% • keystroke hint`) joined by a styled separator (`t.Header.Separator •`), then `ansi.Truncate(result, availWidth, "…")` so it always fits the rectangle.
- `styles/styles.go` `Header` struct — each header part is its own semantic token (`Keystroke`, `KeystrokeTip`, `WorkingDir`, `Percentage`, `Separator`).

**aimux target (evidence):**

- No header on non-dashboard views; the switch stepper is the only "where am I" affordance and only lives in the switch flow (`model.go` `View()` cases ~1331-1360 prepend `renderSwitchStepper()` to two views only).
- aimux already has the breadcrumb *concept* in `switchStepLabels` (`model.go:31-37`) and `viewType` names — it just doesn't render a consistent title bar.

**Effort:** **M.** Add a `renderHeader()` that emits one line: `aimux · <view title>` (and, inside the switch flow, the current step). Draw it at the top of every `View()` branch before content. Reserve its height in layout math (ties into A3).

**Benefit:** orientation in deep wizard flows (Switch → Map Models → Review → Confirm) without relying on the dots alone; consistent brand line; natural home for a one-line status (e.g. "3 providers · 1 CLI active", which `renderDashboardSummary` already computes at `model.go:1145-1185`).

### A3 — Layout discipline: propagate width **and** height; correct frame math; full bg fill

**Why now (correctness):** this is the pattern crush is most rigorous about, and aimux violates it in three measurable ways.

**crush source (evidence):**

- `model/ui.go:2674-2695` — `updateLayoutAndSize()` runs a **two-pass** reconcile: size from textarea height → `SetWidth` may change height → regenerate layout once more.
- `model/ui.go:2755-2900` — `generateLayout()` carves the screen into `image.Rectangle`s with `layout.Vertical/Horizontal`, **subtracting** header height, help height, app margins (1px borders) up front. Every component receives its rect.
- `model/ui.go:2393-2416` — `View()` sets `v.BackgroundColor = m.com.Styles.Background` on the **whole** view, then trims trailing spaces per line. AGENTS.md "Common Gotchas": *"Always account for padding/borders in width calculations."*
- `model/ui.go:2284-2295` — components are told their own rect: `m.chat.Draw(scr, layout.main)`, `editor.Draw(scr, layout.editor)`.

**aimux problems (evidence):**

1. **`m.height` is captured but never used.** `model.go:249` stores `msg.Height`; grep shows it is read nowhere in rendering. Layout decisions ignore terminal height entirely.
2. **Forms get width but not height.** `setForm` (`model.go:217-222`) calls only `.WithWidth(m.width)`. Heights are hardcoded magic numbers: `forms.go:177`, `forms.go:215`, and `model.go:702/759/791` all do `.WithHeight(10)` regardless of terminal size → tall forms scroll unpredictably, short forms waste space.
3. **No full-screen bg fill / no height accounting.** `m.padded()` (`model.go:227-235`) applies `Width(m.width).Padding(1,2)` + bg, but no `Height(m.height)` → on a tall terminal the painted region ends at the content and the lower "corner" shows default terminal bg (the exact artifact crush avoids with `View().BackgroundColor` + rect math).
4. **Fragile frame math.** `switchConfirmationView` dry-run computes `Width((m.width-10)/2 - 2)` twice (`model.go` ~1380, ~1410) with a magic `-10`; adding a header/footer (A1/A2) without centralizing this breaks immediately.

**Effort:** **S–M.** Introduce a tiny `layout` struct `{headerH, footerH, bodyW, bodyH int}` computed once per `WindowSizeMsg` from `m.width/m.height`; pass `bodyH` to forms via `.WithHeight`; give `m.padded` a height; set the model-level background. No new dependency.

**Benefit:** forms fit the terminal; bg fills the whole screen; adding header/footer doesn't cascade magic-number edits; height-aware layout is a prerequisite for A1/A2.

### A4 — Form UX: split the tall single-group Add/Edit forms into paginated groups

**Note on honesty:** crush does **not** use `huh` (it has its own `dialog` system), so this is an aimux-internal improvement. The transferable idea from crush is **dialog discipline = one focused decision per screen** (each `dialog/*.go` has a title, a content panel, and a help row — see `dialog/dialog.go` `Dialog` interface and `dialog/common.go` `RenderContext`). huh gives you this for free via multiple `*huh.Group`s.

**aimux problem (evidence):**

- `forms.go:21-66` — `NewAddProviderForm` is **one giant group of 6 inputs** (Name, Base URL, API Key, Auth Token, Context Window, Discovery URL). On a short terminal this is a tall scrolling form.
- `forms.go` `NewEditProviderForm` (the `return huh.NewForm(huh.NewGroup(...))` block near the bottom) repeats the same 5–6-field single group.
- Contrast: `forms.go` `NewMapModelsForm` (~line 393-460) **already** creates one `*huh.Group` per env var — i.e. the paginated multi-group pattern exists in the codebase but isn't applied to the Add/Edit forms.

**Effort:** **S–M.** Break `NewAddProviderForm` into 3 groups: *Identity* (Name), *Endpoint* (Base URL, Discovery URL), *Credentials* (API Key, Auth Token, Context Window). Each group becomes its own huh screen with its own title. Do the same for edit.

**Benefit:** focused screens, per-group validation surfacing, far less overwhelming on laptops; reuses a pattern aimux already ships (`NewMapModelsForm`).

### A5 — Semantic status color tokens + dimming consistency

**crush source (evidence):**

- `styles/styles.go` `Status` struct — five severity pairs (`Error/Warn/Info/Update/Success` × `Indicator` + `Message`).
- `common/elements.go:199-240` — `Status()` renders `[icon] title description extra` with `TitleColor`/`DescriptionColor` and `ansi.Truncate`.
- Dimming: `model/pills.go:16-22` `pillStyle()` switches `Focused` vs `Blurred` tokens; same pattern across chat items (`Messages.UserFocused`/`UserBlurred`).

**aimux target (evidence):**

- `theme.go:46-55` palette has `Green` + `Red` only — no `Warn`/Yellow. Fetch-failed (`"error"` status) and genuine errors share one red.
- Status rendered **inline ad hoc**: `table.go:55` builds `statusLabel` with a fresh `lipgloss.NewStyle().Foreground(...)` each iteration instead of a token.
- Dimming is inconsistent: `menu.go:35` `menuInactiveStyle` exists (good), but the provider list has no analogous reusable dim token.

**Effort:** **S.** Add `Warn lipgloss.Color` + a `Status{OK,Warn,Error} func/struct` to `theme.go`; replace the inline `table.go:55` styling with tokens.

**Benefit:** consistent severity vocabulary; one place to retune the palette; "fetch failed" can read as warning rather than hard error.

### A6 — Typed multi-severity toasts with icon + width-truncation

aimux is already close here (`notificationMsg` `model.go:80`, success auto-dismiss 4s, errors persist at `model.go:426-437`). The gaps vs crush are severity granularity and robustness.

**crush source (evidence):**

- `model/status.go:117-140` — per-type `Indicator` + `Message` styles; message joined to one line (`strings.Join(strings.Split(msg,"\n")," ")`) and **truncated** with `ansi.Truncate(msg, avail, "…")`, then padded to fill the row so it paints cleanly over the help layer.
- `util.InfoMsg` carries a `TTL`; `clearInfoMsgCmd(ttl)` (`status.go:141-147`) makes TTL per-message, not a global constant.

**aimux target (evidence):**

- `model.go:1195-1208` — only two styles: `notifOKStyle` (green) / `notifErrStyle` (red). No warn/info distinction.
- Hardcoded `4*time.Second` (`model.go:431`) for all success messages.
- No truncation: a long `fmt.Sprintf("Apply failed: %s", err)` (`model.go:362`) can wrap and distort the bar.

**Effort:** **S.** Extend `notificationMsg` with a `severity` enum + optional `TTL`; render an indicator icon; truncate with `ansi.Truncate` (or `lipgloss.Width`-aware truncation) to `m.width`.

**Benefit:** warn vs error vs info; long errors stay one line; per-message TTL (e.g. quick "copied" = 2s, "applied" = 4s).

### A7 — Spinner renders in chrome, not by blanking the whole view

**aimux problem (evidence):** `model.go:1271-1278` — when `m.loading`, `View()` returns *only* `m.padded(spinner + msg)`, replacing the entire screen with "Working…". The provider list / form behind it disappears.

**crush source (evidence):** the todo spinner (`ui.go` `todoSpinner`, `model/pills.go` `renderPills`) renders **inline** beside the affected pill; the chat/list behind it stays visible. `styles.SpinnerIcon` is a fallback glyph.

**Effort:** **S.** Keep the underlying view rendered and overlay the spinner in the footer/header (lands naturally once A1's footer exists).

**Benefit:** context isn't lost during `RetryFetch` / `TestConnectivity` / `Apply` (`model.go:345-410`); user still sees the provider they're testing.

## SKIP (chat-app specific — does not apply to a config manager)

Verified present in crush and correctly out of scope for aimux:

- **Chat message list / streaming render** — `model/chat.go`, `chat/messages.go`, `chat/assistant.go`; lazy `list.List` with ID maps. aimux has no conversation surface.
- **Bottom textarea editor** — `ui.go` `textarea`, `renderEditorView`, `model/ui.go:82-92` height constants. aimux input is `huh` forms, not a prompt box.
- **Session sidebar** — `model/sidebar.go:115-200` (`drawSidebar`: session title, working dir, model info, files/LSP/MCP/skills). The dynamic height allocator `getDynamicHeightLimits` (`sidebar.go:95-110`) is elegant but solves a problem aimux doesn't have. (A provider/CLI list is a *different* feature — not the session sidebar.)
- **Pills (todo progress / queue)** — `model/pills.go` entirely; agent-task concept, no analog.
- **Diff / code rendering** — `diffview/`, `chat/file.go`, `common/diff.go`. aimux's `switchConfirmationView` shows a *config* preview, not source diffs.
- **Image rendering** — `image/` (Kitty graphics), `chat.go` `KittyGraphicsEvent` handler.
- **LSP / MCP / skills resource lists** — `sidebar.go` lsp/mcp/skills sections, `model/lsp*.go`.
- **Completions popup / attachments / onboarding/landing lifecycle / gradient logo (catwalk)** — `completions/`, `attachments/`, `uiOnboarding/uiLanding`, `logo/` gradient helpers.

## EVALUATE (judgment call)

### E1 — Rectangle-based screen-buffer layout (ultraviolet)

**What crush does:** `View()` (`ui.go:2393-2416`) builds a `uv.NewScreenBuffer(w,h)`, `Draw()` (`ui.go:2271-2370`) paints components into `image.Rectangle`s, dialogs draw last to overlay (`ui.go:2360`). AGENTS.md calls this the "hybrid rendering" pipeline.

**Verdict: adopt the *model*, not the library.** `ultraviolet` is Charm-internal and unpublished — aimux cannot import it. But the **idea** — precompute a layout struct of rectangles, pass each to its view, separate "where" from "what" — is already recommended in **A3** and can be done with a 10-line Go struct + lipgloss `JoinVertical/Horizontal`. Do **not** attempt to mirror `uv.Screen`/`layout.Split`/`Draw(scr, rect)`; there's no payoff without the buffer, and it would fight huh's string-based forms.

### E2 — Stacked dialog/overlay system

**What crush does:** `dialog/dialog.go:78-230` — `Overlay` stack with `OpenDialog/CloseFrontDialog`, `DrawCenter` centering, and a clever input **grace period** (`dialog.go:100-180`, `graceQuietPeriod`/`graceMaxDelay`) so async-opened dialogs don't eat in-flight keystrokes.

**Verdict: defer unless aimux grows custom modals.** aimux currently covers its modal needs with `huh` (`NewDeleteConfirmForm`, selects). A full overlay stack is over-engineering today. **However**, a *minimal* centered overlay is the cleanest way to resurrect aimux's dead `?` help screen (Appendix) without dedicating a full view to it — so revisit if A1's footer-toggle isn't enough or if a "confirm destructive action with context" modal is wanted.

## Recommended quick-win sequence

1. **A3 (layout struct + width/height propagation + bg fill)** — unblocks the rest; pure correctness.
2. **A1 (footer `help.Model`)** — biggest perceived polish jump *and* deletes dead code (Appendix). Needs A3's footer-height reservation.
3. **A2 (header bar)** — natural follow-on; needs A3's header-height reservation.
4. **A5 (semantic tokens)** — cheap, do alongside A1/A2 while touching `theme.go`.
5. **A4 (form grouping)** — independent; ship whenever forms are touched.
6. **A6 → A7** — small refinements once the footer exists.

E1 is folded into A3. E2 stays parked.

## Appendix — dead-code finding: the `?` help overlay does nothing

**Evidence (authoritative, current `main`/branch `feat/enrich_clis_integrations`):**

- `model.go:98` — `Help: key.NewBinding(key.WithKeys("?"), key.WithHelp("?", "help"))` is in `menuKeys`.
- `model.go:178` — `showHelp bool // when true, render help overlay instead of current view`.
- `model.go:412-419` — `Update`/`handleKeyMsg`: `?` sets `m.showHelp = true`; any subsequent key sets it `false`.
- `model.go:1990` — `func (m *model) renderHelpOverlay() string { … }` is fully implemented.
- **But:** `grep renderHelpOverlay internal/tui/*.go` → the function is **defined only, never called**. `showHelp` is **set** at 412/413/419 and **read nowhere in `View()`**.

**Result:** pressing `?` toggles a flag that no render path consults. The advertised `?` help binding is a no-op. This is fixed for free by A1 (drive a real `help.Model` footer + full-help toggle) — or, minimally, by adding `if m.showHelp { return m.padded(m.renderHelpOverlay()) }` at the top of `View()`.
